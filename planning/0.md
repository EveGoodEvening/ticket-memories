下面是我基于当前仓库状态、MacRumors 背景、以及 Apple/Spotify 公开 API 约束整理的详细规划。当前 repo 是一个空 Git 仓库：只有 `.git`，没有 iOS 工程、没有现有代码、没有提交记录，所以这份计划按“从 0 开始设计 iOS app”来写。

## 1. 我对产品的理解

**Ticket Memories：票根回忆**不是一个单纯的“Wallet pass 创建器”，而是一个把用户过去参加过的演唱会、展览、赛事、旅行活动等门票整理成“人生时间线”的回忆产品。

核心价值应该是：

> 用户不只是保存票，而是在回顾“我去过哪里、见过谁、听过什么、当时发生了什么”。

背景文章里 iOS 27 可能会加入原生自定义 Wallet pass 创建能力，这对产品方向有两个影响：

1. **不要把主价值押在“创建 pass”上**  
   这部分很可能被 Apple 原生功能 commoditize。

2. **应该把主价值押在“pass 之后的记忆整理、分享、回顾、印刷”上**  
   Apple 可以帮用户把票放进 Wallet，但不一定会帮用户做年度回顾、照片关联、Spotify 歌单、纪念册、人生地图。

所以我建议产品定位是：

> Wallet 是入口，Ticket Memories 是记忆层。

## 2. 关键平台约束

这里是最重要的技术现实。

### 2.1 Wallet 数据不能假设可以“全量自动读取”

公开 PassKit 能处理 `PKPass`、添加 pass、读取 pass 的部分展示信息、打开 Wallet URL 等。但第三方 app 不应该把“自动扫描用户 Wallet 里所有历史门票”当成唯一入口，因为：

- Apple 对 Wallet、支付卡、证件、Secure Element pass 有严格限制。
- iOS 27 的“Create a Pass”目前仍是传闻，是否开放 API 未知。
- 即使用户有很多 Wallet pass，也要验证 app 实际可读取范围。
- App Store 审核和隐私预期上，也不适合表现得像在“爬取用户 Wallet”。

### 2.2 推荐的导入策略

不要只做一个入口，应该做多入口：

1. **PassKit / Wallet 可访问 pass 导入**
   - 读取可访问的非敏感 pass 信息。
   - 提取标题、组织方、时间、地点、pass icon、pass URL 等。

2. **`.pkpass` 文件导入**
   - 用户从邮件、文件、网页、聊天工具分享 `.pkpass` 到 app。
   - 这是非常重要的 fallback。
   - 如果拿到原始 `.pkpass`，可以解析 `pass.json`、图片资源、location、barcode metadata 等。
   - 但不要上传 authentication token 或敏感 barcode 内容。

3. **QR code / barcode 扫描**
   - 用户可以扫描旧票、电子票截图、纸质票。
   - 初期只把它作为“创建记忆”的辅助输入，不一定要重新生成 Wallet pass。

4. **手动创建记忆**
   - 标题、日期、地点、照片、笔记。
   - 这是必须有的兜底能力。

5. **照片/日历辅助推荐，后期做**
   - 根据活动日期和地点推荐照片。
   - 根据日历事件猜测活动。
   - 这些都涉及权限，建议放到 v1.5+。

## 3. 推荐产品 wedge

不要一开始覆盖所有 Wallet pass，比如会员卡、礼品卡、登机牌、优惠券都做。范围太散。

我建议第一版聚焦：

> **演唱会 / 音乐节 / 展览 / 剧场 / 体育赛事这些“有情绪价值的活动票”。**

原因：

- 用户更愿意为回忆、分享、纪念册付费。
- Spotify 歌单天然适配演唱会和音乐节。
- 地图和年度回顾更有视觉冲击。
- 纪念册/海报/票根打印更容易变现。
- 比会员卡、购物券更符合“人生时间线”。

## 4. MVP 功能范围

### MVP 必须做

1. **导入或创建一张活动记忆**
   - 从 `.pkpass` 导入。
   - 手动创建。
   - 可选：尝试从 PassKit 读取可访问 pass。

2. **活动详情页**
   - 标题。
   - 日期。
   - 场馆 / 城市。
   - 地图位置。
   - pass 缩略图。
   - 照片 / 视频。
   - 笔记。
   - Spotify playlist 链接。

3. **时间线**
   - 按年月展示活动。
   - 支持年份筛选。
   - 支持卡片式浏览。

4. **地图**
   - 地图上显示活动地点。
   - 点开 marker 进入活动详情。

5. **年度回顾**
   - 生成一个静态 recap 页面或图片：
     - 去了多少场活动。
     - 去过多少城市。
     - 最常去的场馆。
     - 年度精选照片。
     - 年度 playlist。
   - MVP 先做图片/PDF，不做复杂视频。

6. **分享**
   - 使用系统 Share Sheet 分享：
     - 单个活动卡片。
     - 年度回顾图片。
     - 简单 PDF。
   - MVP 不建议一开始做公开网页后端。

7. **基础付费点**
   - 免费主题 + 高级主题。
   - 本地免费，云同步订阅后置。
   - 打印先做“导出 PDF”，实际下单放 v1/v2。

### MVP 暂缓

这些很有吸引力，但不建议一开始做：

- 完整实体纪念册下单系统。
- AI 自动写长篇回忆文案。
- 视频版年度回顾。
- 自动扫描全部 Wallet 历史。
- 社交关系链。
- 多人协作相册。
- 复杂 pass 重新生成器。
- 全球所有票务平台集成。

## 5. 推荐技术栈

### iOS 客户端

| 领域 | 推荐 |
|---|---|
| UI | SwiftUI |
| 本地数据 | SwiftData |
| 地图 | MapKit for SwiftUI |
| 照片选择 | PhotosUI / PhotosPicker |
| 视频预览 | AVKit |
| pass 处理 | PassKit |
| 文件导入 | DocumentPicker / fileImporter |
| 分享 | ShareLink / UIKit share sheet |
| PDF/图片导出 | SwiftUI renderer + PDFKit/CoreGraphics |
| 订阅/IAP | StoreKit 2 |
| 登录/令牌 | Keychain |
| 同步 | CloudKit private database |
| 后端，后期 | Cloudflare Workers / Supabase / Firebase / custom backend |

### 为什么先用 CloudKit

CloudKit 对这个产品很适合，因为：

- iOS 原生。
- 用户不用创建新账号。
- 适合私密个人数据同步。
- App Store 审核和用户信任更好。
- 对“回忆、照片、票根”这种私密数据更自然。

但 CloudKit 不适合全部事情：

- 公开分享网页不方便。
- 打印订单不适合直接靠 CloudKit。
- 跨平台 web 产品受限。

所以推荐架构是：

> **本地优先 + CloudKit 私有同步 + 后期独立 backend 负责分享页和实体商品。**

## 6. 三种架构方案对比

### 方案 A：纯本地 iOS app

**做法**

- SwiftData 存本地。
- 照片/视频复制到 app sandbox。
- 分享只导出图片/PDF。
- 不做账号、不做云同步、不做公开网页。

**优点**

- 开发最快。
- 隐私最强。
- App Store 审核风险低。
- MVP 验证非常快。

**缺点**

- 换手机/多设备体验差。
- 很难做订阅。
- 公开分享页和打印不方便。

**适用阶段**

- 原型。
- 第一个 TestFlight。
- 产品验证。

---

### 方案 B：iOS + CloudKit

**做法**

- SwiftData + CloudKit private database。
- 用户数据跟随 iCloud。
- 高级同步作为订阅功能。
- 分享仍以图片/PDF 为主。

**优点**

- 保持 Apple-native。
- 隐私感强。
- 不需要自己维护账号系统。
- 很适合个人回忆类 app。

**缺点**

- 非 Apple 平台弱。
- 公开网页分享能力有限。
- CloudKit 调试和数据模型迁移要谨慎。

**适用阶段**

- MVP。
- 付费同步。
- 长期个人资料库。

---

### 方案 C：iOS + 自建后端

**做法**

- 自建用户系统或 Sign in with Apple。
- 服务器保存 share page、print order、media derivatives。
- 后端生成公开网页和打印 PDF。

**优点**

- 分享页体验最好。
- 打印业务可控。
- 方便后续 web 版、Android 版。
- 方便做增长和 referral。

**缺点**

- 成本更高。
- 隐私和安全责任更重。
- 开发周期更长。
- 需要维护媒体存储、权限、删除、订单、支付。

**适用阶段**

- v1 之后。
- 已验证用户愿意分享/打印。
- 开始做实体纪念册收入时。

### 推荐

先做：

> **方案 A → 方案 B → 方案 C**

不要从第一天就做完整后端。这个产品的最大不确定性不是技术，而是用户是否真的愿意持续整理 ticket memories、是否愿意付费打印或升级主题。先验证这个行为。

## 7. 核心用户流程

### 7.1 首次打开

1. 欢迎页解释价值：
   - “把你的票根变成人生地图。”
   - “整理演唱会、展览、赛事和旅行活动。”
2. 隐私说明：
   - 数据默认保存在设备上。
   - 不上传票据敏感信息。
   - 照片由用户主动选择。
3. CTA：
   - 导入 Wallet/pass。
   - 手动创建第一段回忆。

### 7.2 创建记忆

用户有三种入口：

1. 导入 `.pkpass`
2. 扫描 QR / 上传截图
3. 手动创建

导入后进入“确认信息”页：

- 标题：Taylor Swift — Tokyo Dome
- 日期：2025-02-08
- 场馆：Tokyo Dome
- 城市：Tokyo
- 类型：Concert
- pass 图片
- 是否保存 barcode：默认不保存或只保存加密摘要
- 添加照片
- 添加笔记
- 添加 Spotify playlist

### 7.3 时间线浏览

首页建议是时间线，而不是地图。

原因：

- “回忆”更自然按时间看。
- 地图是探索视角，不是每天打开的主视角。
- 年度回顾也基于时间线。

结构：

- Today / This Year / Past Years。
- 年份分组。
- 月份分组。
- 大卡片展示代表图、地点、标题。

### 7.4 地图浏览

地图展示：

- 所有 event pins。
- 按年份筛选。
- 按类型筛选。
- 城市 cluster。
- 点击 marker 展示小卡片。

MVP 不需要复杂路线动画，后期可以做：

- 年度足迹路线。
- 城市热力图。
- “我看演唱会去过的世界地图”。

### 7.5 年度回顾

用户选择年份后生成：

- 年度标题。
- 活动总数。
- 城市数量。
- 最常去场馆。
- 最远一次活动。
- 精选照片 grid。
- Spotify 年度 playlist。
- 地图缩略图。
- 可分享长图或 PDF。

后期可以做类似 Spotify Wrapped 的 Story 体验。

## 10. 关键功能设计

### 10.1 Wallet/pass 导入

导入流程：

1. 用户点“Import Ticket”。
2. app 显示可用入口：
   - From Wallet
   - From File
   - Scan QR
   - Create Manually
3. 如果是 `.pkpass`：
   - 使用 `PKPass(data:)` 验证可解析。
   - 尝试读取 `localizedName`、`organizationName`、`relevantDate`、icon。
   - 如果能访问原始包内容，解析 `pass.json` 中的：
     - eventTicket / generic 类型。
     - relevant date。
     - locations。
     - auxiliary/back fields。
4. 进入确认页。
5. 用户确认后才保存成 `MemoryEvent`。

重要 UX：

- 不要自动把所有导入内容直接入库。
- 一定要给用户编辑机会。
- 时间、地点识别经常会错。

### 10.2 地点识别

地点来源优先级：

1. pass 里明确的 location 坐标。
2. pass 字段里的 venue/address。
3. 用户手动输入场馆。
4. MapKit local search。
5. 用户手动拖 pin。

需要保存 `locationConfidence`：

- exact
- inferred
- manual
- unknown

这样 annual recap 可以避免用错误地点生成奇怪地图。

### 10.3 照片/视频关联

MVP：

- 用户主动选择照片/视频。
- app 生成缩略图。
- 原图可选压缩保存。
- 视频只生成 poster frame 和本地引用。

v1.5：

- 请求 limited photo library access。
- 根据活动日期 ±1 天推荐照片。
- 根据照片 GPS 和 venue 距离推荐照片。

注意：

- 不要第一次启动就请求 Photos 权限。
- 在用户点“Add Photos”时再请求。
- 默认用 PhotosPicker 可以减少权限压力。

### 10.4 Spotify 歌单

分两阶段。

#### MVP：链接模式

用户粘贴 Spotify playlist URL，app 保存：

- playlist URL
- playlist name，可选通过公开 metadata 获取
- cover image，可选
- 打开 Spotify app

优点：

- 不需要一开始做 OAuth。
- 上线快。
- 隐私解释简单。

#### v1：OAuth 模式

使用 Spotify OAuth with PKCE：

- 读取用户 playlist 列表。
- scope 使用最小权限：
  - public playlist 可以不需要太多权限。
  - 私有 playlist 需要 `playlist-read-private`。
- token 存 Keychain。
- 不做播放控制，除非真的需要。

不建议 MVP 做 Spotify 播放控制，因为播放控制通常需要额外 scope，且部分能力要求 Premium。

### 10.5 年度活动回顾

MVP 生成静态 recap：

- SwiftUI view 渲染成长图。
- 可导出 PNG/JPEG。
- 可导出 PDF。
- ShareLink 分享。

模板：

1. Minimal Timeline
2. Concert Wrapped
3. Passport Map
4. Ticket Wall
5. Photo Diary

高级主题可以作为付费点。

### 10.6 可分享页面

分阶段。

#### MVP

只做本地导出：

- 单活动分享卡片。
- 年度回顾长图。
- PDF。

#### v1

做公开 share page：

- 用户点“Publish”。
- 上传经过处理的公开数据。
- 生成不可猜测 URL。
- 支持取消发布。
- 支持隐藏地点精度。
- 支持只显示城市，不显示具体场馆。

#### v2

做社交和增长：

- “My 2026 Concert Map”
- “Top 10 venues”
- “Festival yearbook”
- referral watermark
- 可嵌入 Instagram/TikTok story 的竖版模板

### 10.7 实体纪念册

不建议 MVP 直接做完整下单。

推荐路线：

#### 阶段 1：Print-ready PDF

用户选择：

- 年份。
- 主题。
- 活动。
- 照片。
- 页面尺寸。

app 生成 PDF：

- 封面。
- 年度地图。
- 时间线。
- 每个活动一页。
- QR code 链接到 Spotify playlist 或 share page。

#### 阶段 2：手动 fulfillment 验证

先小规模验证：

- 用户提交 PDF。
- 你人工或半自动找打印供应商下单。
- 验证用户是否愿意付费、客单价、退货率。

#### 阶段 3：接入 print-on-demand

后期再接：

- 纪念册。
- 海报。
- 明信片。
- 票根贴纸。
- framed print。

实体商品属于非数字实物，支付规则通常不同于 app 内数字内容，但上线前仍要核对最新 App Store Review Guidelines。数字主题、云同步、数字导出高级功能则应走 StoreKit 2 / IAP。

## 11. 变现设计

### 免费版

- 创建有限数量 memories，比如 20 个。
- 基础时间线。
- 基础地图。
- 基础分享图。
- 1-2 个免费主题。

### Pro 订阅

适合月付/年付：

- 无限 memories。
- iCloud sync。
- 高级年度回顾。
- 高级主题。
- 批量导入。
- 高清导出。
- 视频 recap，后期。
- 隐私分享页管理，后期。

### 一次性购买

适合：

- 单个高级主题包。
- 单次年度 PDF 导出。
- 单个纪念册模板包。

### 实体商品

高毛利方向：

- 年度纪念册。
- 演唱会海报。
- 票根墙 poster。
- 城市地图 print。
- 明信片套装。
- festival scrapbook。

### 推荐优先级

1. 高级主题。
2. 高清年度回顾导出。
3. Cloud sync。
4. Print-ready PDF。
5. 实体纪念册。

不要一开始把收入核心压在实体商品上，因为供应链、客服、物流、退货、质量控制都会拖慢产品验证。

## 12. 隐私与安全策略

这个产品会处理非常私密的数据：

- 用户去过哪里。
- 什么时间在某个场馆。
- 和哪些照片关联。
- 票据二维码可能包含可入场凭证。
- Spotify 品味。
- 旅行和生活轨迹。

所以必须从第一版就明确：

### 默认原则

- 默认本地保存。
- 默认不公开。
- 默认不上传原始 ticket barcode。
- 默认不上传原始 pass authentication token。
- 分享前明确预览。
- 公开页面可撤销。
- 地点可模糊到城市级别。
- 删除 account/data 要完整。

### 权限请求

只在需要时请求：

- Photos：用户点添加照片时。
- Location：不一定需要持续定位；地图展示不需要定位权限。
- Cloud：用户开启同步时。
- Spotify：用户选择连接 Spotify 时。

### App Store 文案

不要说：

- “自动备份你的 Wallet”
- “复制你的 Wallet”
- “保存所有票据凭证”

应该说：

- “Organize event memories from tickets and passes you choose to import.”
- “Your memories stay private unless you share them.”
- “Sensitive ticket credentials are not uploaded by default.”

## 15. 最大风险与应对

### 风险 1：Wallet pass 读取能力不够

应对：

- `.pkpass` 文件导入。
- 手动创建。
- QR scan。
- 截图/OCR 后期。
- 不把“自动读取 Wallet 全部历史”作为核心承诺。

### 风险 2：Apple iOS 27 原生功能覆盖 pass creation

应对：

- 不做 pass maker。
- 做记忆整理、地图、照片、Spotify、recap、打印。
- Apple 做入口，Ticket Memories 做后续生命周期。

### 风险 3：用户懒得整理

应对：

- 第一张 memory 创建必须极简。
- 默认用 pass 信息自动填。
- 照片推荐后期提高完成度。
- 年度 recap 提供即时奖励。
- 分享图让用户有动机补全内容。

### 风险 4：实体纪念册运营复杂

应对：

- 先 PDF。
- 再人工 fulfillment。
- 最后接 print-on-demand。
- 不要一开始做完整电商系统。

### 风险 5：隐私担忧

应对：

- 本地优先。
- 清晰隐私文案。
- 不上传敏感 ticket credential。
- 分享可预览、可撤销。
- 地点可模糊。

## 16. 我建议的首版路线

如果现在要开始，我建议第一阶段目标非常明确：

> 做一个本地-first iOS app，让用户可以导入或手动创建 10 张演唱会/展览票根，自动形成时间线和地图，并生成一张漂亮的年度回顾分享图。

首版最小闭环：

1. 创建/导入活动。
2. 补充照片和笔记。
3. 在时间线和地图里看到它。
4. 生成年度回顾。
5. 分享给朋友。

只要这个闭环有用户喜欢，后面的高级主题、云同步、纪念册都有空间。

## 17. 需要你后续决定的问题

1. 第一版是只做英文，还是中英双语？
2. 首批用户更偏演唱会，还是展览/旅行/体育也一起覆盖？
3. 你希望 MVP 优先做漂亮 UI，还是优先做 pass 导入准确率？
4. 分享页是否必须第一版就有公开 URL，还是先用图片/PDF？
5. 实体纪念册是长期愿景，还是你希望尽早验证？
6. 是否接受第一版不承诺“自动读取所有 Wallet 门票”？

我的建议答案是：

- **先做中英架构可扩展，但 MVP UI 可先英文或中文单语。**
- **先聚焦演唱会/音乐节。**
- **优先做漂亮 timeline + recap，导入用 `.pkpass`/手动兜底。**
- **分享先做图片/PDF，不做公开 URL。**
- **实体纪念册先做 print-ready PDF。**
- **不要承诺自动读取所有 Wallet。**
